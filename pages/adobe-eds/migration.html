Automate Migration

Learn how to migrate content and data from existing systems to Adobe Edge Delivery Services using automation tools and best practices.

Migration Overview

Migrating to Adobe Edge Delivery Services involves transferring content, assets, and configurations from your existing platform. Modern tools and APIs make this process automated and efficient.

Migration Strategy: Plan your migration in phases - assess source systems, extract data, transform to new format, validate, and deploy with zero downtime.

Pre-Migration Assessment

Before starting migration, evaluate your current environment:

Content Audit

  • Inventory - Document all content types and volume
  • Structure - Map existing content model to target model
  • Dependencies - Identify linked content and relationships
  • Assets - Catalog images, videos, and other media
  • Metadata - Review SEO and custom data fields

System Analysis

  • Source Platform - Current CMS/database system
  • APIs Available - Extraction capabilities
  • Authentication - Access requirements and credentials
  • Performance - Expected migration duration
  • Risks - Identify potential issues and blockers

Migration Tools & Solutions

Adobe Migration Tool

Official tool for migrating content to AEM and Edge Delivery Services:

npm install -g @adobe/aem-migration-tool # Initialize migration project aem-migrate init --source wordpress --target aem-edge

Custom Migration Scripts

Build custom migration logic for unique requirements:

// migration-script.js const axios = require('axios'); const { AEMClient } = require('@adobe/aem-api'); async function migrateContent() { const sourceClient = new SourceClient({ baseUrl: 'https://old-site.com/api' }); const aemClient = new AEMClient({ org: 'your-org', accessToken: process.env.AEM_TOKEN }); // Extract content from source const pages = await sourceClient.getPages(); // Transform and load to AEM for (const page of pages) { const transformed = transformPage(page); await aemClient.createAsset(transformed); } } function transformPage(page) { return { title: page.name, path: `/content/pages/${page.slug}`, content: page.body, metadata: { description: page.meta_description, keywords: page.tags } }; }

Extracting Data from Source Systems

Database Export

For database-driven systems, export data in standard formats:

# Export from WordPress database mysqldump -u wordpress -p wordpress_db posts > posts.sql # Convert to JSON for easier processing node convert-sql-to-json.js posts.sql > content.json

API-Based Extraction

Use REST APIs to pull content from source systems:

// Extract content via REST API const axios = require('axios'); async function extractPages() { try { const response = await axios.get( 'https://source-api.com/api/v1/pages', { headers: { 'Authorization': `Bearer ${process.env.SOURCE_API_KEY}` } } ); // Save extracted data const fs = require('fs'); fs.writeFileSync( 'extracted-pages.json', JSON.stringify(response.data, null, 2) ); console.log(`Extracted ${response.data.length} pages`); } catch (error) { console.error('Extraction failed:', error.message); } } extractPages();

Data Transformation

Transform extracted data to match your target content model:

Content Mapping

Map source fields to target structure:

// Transform content to target format function transformContent(sourceContent) { return { // Target field mappings title: sourceContent.post_title, slug: sourceContent.post_name, description: sourceContent.post_excerpt, content: cleanHTMLContent(sourceContent.post_content), author: sourceContent.post_author, publishDate: sourceContent.post_date, // Custom field mappings seoTitle: sourceContent.meta['_yoast_title'] || sourceContent.post_title, seoDescription: sourceContent.meta['_yoast_meta_description'], // Media references featuredImage: sourceContent.featured_image_url }; } function cleanHTMLContent(html) { // Remove unsupported HTML, sanitize, normalize return html.replace(/<script[^&]*<\/script>/gi, ''); }

Validation & Testing

Ensure migrated content is accurate and complete:

Data Validation Checks

  • Completeness - All required fields are populated
  • Data Types - Fields match expected format
  • Referential Integrity - Links between content are valid
  • Asset References - All media files are accessible
  • Duplicates - No duplicate content created

Testing Strategy

// Validation script async function validateMigration() { const validationRules = { requiredFields: ['title', 'slug', 'content'], dataTypes: { publishDate: 'date', tags: 'array', metadata: 'object' } }; const results = { passed: 0, failed: 0, errors: [] }; for (const page of migratedPages) { // Check required fields validationRules.requiredFields.forEach(field => { if (!page[field]) { results.failed++; results.errors.push(`Page ${page.id} missing ${field}`); } else { results.passed++; } }); } return results; }

Migration Deployment

Deploy migrated content to Edge Delivery Services:

Phase-Based Deployment

  1. Staging - Deploy to staging environment for testing
  2. Validation - Verify all content displays correctly
  3. User Testing - Get stakeholder sign-off
  4. Production - Deploy to live environment
  5. Monitoring - Monitor for issues post-migration

Deployment Command

# Deploy migration aem-migrate deploy \ --env production \ --source-batch extracted-pages.json \ --verify-assets \ --post-migration-hooks

Post-Migration Tasks

  • Redirect Setup - Configure URL redirects from old to new paths
  • SEO Optimization - Verify metadata and update search console
  • Performance Review - Monitor page speed and optimize
  • User Training - Educate content teams on new platform
  • Documentation - Update internal documentation
  • Monitoring - Set up alerts and analytics tracking

Common Migration Issues

Asset Migration Failures

Verify source URLs are accessible and properly referenced in content. Use batch retry mechanisms for failed assets.

Content Formatting Issues

Clean HTML/rich text content to remove unsupported tags. Use HTML parsers to normalize formatting.

Performance During Migration

Execute migrations during off-peak hours. Use batch processing with configurable concurrency limits.

Migration Best Practices

  • Dry Run First - Always test with a subset before full migration
  • Backup Source - Keep backups of original content
  • Monitor Logs - Track all migration activities and errors
  • Schedule Carefully - Plan for content freeze during migration
  • Document Process - Record migration steps for future reference
  • Rollback Plan - Have a strategy to revert if issues occur
