Adobe Commerce Overview

Comprehensive introduction to Adobe Commerce, from core architecture to advanced features. Everything you need to understand this powerful e-commerce platform.

What is Adobe Commerce?

Adobe Commerce (formerly Magento) is an enterprise-grade e-commerce platform that powers some of the world's leading online retailers. It's built on a modular architecture that allows complete customization and scalability.

Key Features: Advanced product catalogs, flexible promotions, multi-channel selling, sophisticated analytics, API-first architecture, progressive web application (PWA) support, and enterprise security.

Core Architecture

As an AEM developer, you need to understand two interconnected systems: AEM (content management) and Adobe Commerce/Magento (e-commerce backend). The CIF Add-on is the bridge between them.

Key Architecture Points:
Author Workflow: Editor opens Commerce tile → CIF queries Magento via GraphQL → Editor selects products → Author enriches with copy/layouts → Publishes to Publisher

Live Website (Publisher): Visitor loads product page → Page content is (authored & cached) → CIF queries Magento in real-time for current prices/inventory → Visitor sees live product info

Critical Difference: Author sees cached product data. Publisher always shows fresh data from Magento to visitors via CIF.

Key Tech Details (For AEM Developers)

▸ GraphQL is the API Magento exposes a GraphQL API that CIF uses to fetch data. You don't write REST calls; CIF handles the GraphQL.
▸ CIF is JavaScript-based The CIF client library (running in AEM) makes GraphQL requests to Magento. You never talk to Magento directly from AEM code.
▸ No database connections needed AEM doesn't connect to Magento's MySQL database. Everything goes through the GraphQL API. This is HUGE for security and scalability.
▸ Real-time data Every time an author browses products, CIF queries live data. Prices, inventory, and availability are always current.
▸ Caching layer AEM can cache GraphQL responses to reduce calls to Magento, improving performance.

🔄 What Happens When Magento Data Changes?

After an author creates a page with product data in AEM, what happens if that product's price or inventory changes in Magento?

Scenario: Visitor Loads Page
1. Visitor opens product page on Publisher
The page already has the product title, description, image that the author added

2. CIF client-side code runs
It makes a fresh GraphQL query to Magento for current price & inventory

3. Real-time data replaces cached data
Price, stock status, discounts are ALWAYS fresh from Magento

Result: ✓ Visitor always sees current pricing and stock
Scenario: Author Updates Page
1. Author clicks on product component
Wants to check current product data

2. CIF re-queues Magento
Makes a fresh GraphQL call to get latest product info

3. Data shown to author is refreshed
Author sees: new price, new stock, new description if anything changed

Result: ⚠️ Author makes manual decision to update the page
Key Insight: Content vs. Data
  • Authored content stays the same: Copy, images, layouts, CTAs that the author wrote in AEM don't change
  • Live product data refreshes: Every time a visitor lands on the page, it queries Magento for current price, inventory, availability
  • Author sees staleness: If the author comes back days later, they see old data because CIF only refreshes when explicitly browsing. But visitors always get fresh data.
  • No automatic re-publishing: If product gets discontinued or price drops 50%, pages don't auto-update. Author needs to manually check and make decisions.

Real-World Example: Price Drop

Author's Perspective
Day 1: Creates page with "Laptop - $999"
Day 5: Product drops to $699 in Magento
Day 10: Author opens page in AEM → Still shows "$999" in the authored content
Action: Author must manually update or refresh the product reference
Visitor's Perspective
Day 1: Visitor lands → Sees $999 (correct)
Day 5: Price drops in Magento
Day 10: Visitor lands → CIF queries Magento → Sees $699 (current!)
Result: ✓ Visitor always sees correct price

🛡️ Cache Invalidation Strategy

To keep author content fresh, here's what happens:

▸ On-demand refresh
- Author can manually refresh product data when editing
▸ TTL (Time-to-Live)
- Cached GraphQL responses expire after X minutes/hours and must be re-queried
▸ Magento webhooks
- When a product updates in Magento, it can trigger AEM cache invalidation
▸ Periodic sync
- Scheduled jobs can refresh popular product data to keep it current

Adobe Commerce Backend (Magento) Architecture

While you work in AEM, it's helpful to understand what's happening in Magento behind the scenes. Magento is built on a modular system:

1. Service Layer Architecture

At the heart of Adobe Commerce is a service-oriented architecture that separates concerns into distinct, independently deployable modules. This allows developers to focus on specific features without affecting the entire system.

2. Module System

The platform is built as a collection of interdependent modules, each handling specific functionality:

  • Catalog Module - Manages product data, categories, and attributes
  • Sales Module - Handles orders, invoices, and shipments
  • Customer Module - Manages customer accounts and addresses
  • Payment Module - Integrates payment processing
  • Shipping Module - Manages shipping methods and rates

3. Database Layer

Adobe Commerce uses MySQL/MariaDB to store data in an optimized relational structure. The database schema is designed for performance and scalability with proper indexing.

Request Flow

  1. URL request is received by the web server
  2. Router analyzes the URL and routes to appropriate controller
  3. Controller processes the request and interacts with models
  4. Models fetch data from database or external services
  5. Data is formatted and passed to view/template
  6. HTML response is rendered and sent to browser

Essential Concepts

Models, Views, and Controllers

Adobe Commerce follows the MVC pattern:

  • Models - Represent data and business logic (e.g., Product, Order)
  • Views - Template files that display data (PHTML files)
  • Controllers - Process requests and coordinate M-V interaction

Collections vs. Single Items

Adobe Commerce uses collection classes to manage groups of models efficiently with filtering, sorting, and pagination capabilities.

Plugin System (Interceptors)

Plugins allow you to modify behavior of public methods without changing the original class, following the Open/Closed Principle.

Dependency Injection

Adobe Commerce uses automatic dependency injection to improve testability and reduce coupling between classes.

Adobe Commerce Cloud vs. Self-Hosted

Adobe Commerce is available in two deployment options:

Adobe Commerce Cloud

  • Fully managed infrastructure on AWS
  • Automatic scaling and high availability
  • Built-in CDN and DDoS protection
  • Integrated staging and production environments
  • Enhanced security and compliance

Self-Hosted / On-Premises

  • Complete control over infrastructure
  • Custom server configurations
  • Responsible for security and maintenance
  • Potentially lower costs for large deployments
  • Requires dedicated DevOps resources

Extension Development

Developers extend Adobe Commerce functionality by creating custom modules (extensions). Each module contains:

  • etc/module.xml - Module configuration and dependencies
  • etc/di.xml - Dependency injection configuration
  • Helper classes - Utility functions
  • Models - Data models and repositories
  • Controllers - Request handlers
  • Blocksand Templates - View layer components
  • Database migrations - Schema updates

Development Best Practices

  • Keep Code Modular - Create focused, reusable modules
  • Use Dependency Injection - Avoid hardcoded dependencies
  • Follow PSR Standards - Use PSR-4 autoloading and PSR-2 coding standards
  • Test Your Code - Write unit and integration tests
  • Document Everything - Maintain clear documentation
  • Version Your Extensions - Use semantic versioning
  • Avoid Hacks - Find proper extension points instead of core modifications
